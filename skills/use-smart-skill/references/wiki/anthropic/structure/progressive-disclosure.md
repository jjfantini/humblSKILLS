## The three levels

| Level | What loads                                 | When                                          |
|-------|--------------------------------------------|-----------------------------------------------|
| 1     | YAML frontmatter                           | Always, in the system prompt                  |
| 2     | SKILL.md body                              | When Claude decides the skill is relevant     |
| 3     | Files under `references/`, `scripts/`, `assets/` | Only when Claude chooses to read them    |

This is the core mechanism that keeps skill overhead low: the frontmatter is tiny (loads always), the body is medium (loads on trigger), and the long tail of detail stays on disk until needed.

## Writing for each level

### Level 1 - frontmatter

- Treat every character as if it lives in the system prompt (it does).
- No `<` or `>` (banned); keep under 1024 chars for `description:`.
- Get the trigger design right here - this is where activation happens.

### Level 2 - SKILL.md body

- Core workflow and decision logic. <=5,000 words is the upper bound; ~200 lines is a good target.
- Put critical instructions near the top. Claude reads top-down and weighs earlier content.
- Use headers for scannability: `## When to Use`, `## Examples`, `## Troubleshooting`.
- Link out, don't inline: every time you would copy-paste a reference doc into SKILL.md, write it as a concept file under `references/wiki/` and deep-link.

### Level 3 - linked files

- `references/` - prose documentation loaded on demand (API guides, style docs, historical context).
- `scripts/` - deterministic executables the skill runs instead of trusting language instructions (validation, parsing, batch ops).
- `assets/` - static resources the skill emits (templates, fonts, brand files).

## humblSKILL structure reinforces this

The humblSKILL pattern is explicit progressive disclosure applied recursively:

```
skill-root/
|-- SKILL.md                         # Level 2 - thin router
`-- references/
    |-- _index.md                    # Level 3 - map (regenerated by lint.sh)
    |-- _brain.md                    # Level 3 - spec
    |-- _template.md                 # Level 3 - concept shape
    |-- patterns.md, decisions.md, log.md   # Level 3 - meta memory
    |-- wiki/<context>/<category>/<concept>.md  # Level 3 - atomic concepts
    `-- raw/                         # Level 3 - unfiltered sources
```

Claude reads SKILL.md (Level 2), sees pointers like `Read references/wiki/anthropic/description/trigger-design.md`, then loads only the concepts relevant to the current task.

## Red flags that progressive disclosure is broken

- SKILL.md over 300 lines -> scaffold or migrate the long sections into wiki concepts.
- Instructions repeated across SKILL.md and references/ -> single-source-of-truth in references/, SKILL.md points to it.
- Claude ignoring instructions near the bottom of SKILL.md -> move critical instructions up OR offload to a script (deterministic beats language).
- Slow/degraded responses -> too much content loading; check that references/ files aren't being eagerly read.
