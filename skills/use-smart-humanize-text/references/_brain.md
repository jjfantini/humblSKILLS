# Brain Spec (authoritative)

The brain is the self-learning memory every Smart Skill carries. It makes
each skill traceable (raw sources cited by wiki concepts), transparent
(all artefacts are markdown, human-browsable), and improvable (patterns
and decisions compound session over session).

This file is the complete authoritative spec: territories, protocol,
operations with worked examples, naming, linking, writing principles,
and command routing. `_template.md` carries the wiki concept file shape.

---

## Three Territories

| Territory | Owner | Location                               | Contents                                      |
|-----------|-------|----------------------------------------|-----------------------------------------------|
| Raw       | Human | `references/raw/`                      | Unfiltered sources: articles, PDFs, screenshots, transcripts, exports |
| Wiki      | LLM   | `references/wiki/<ctx>/<cat>/<c>.md`   | Distilled, one-concept-per-file articles with `sources:` backlinks |
| Meta      | LLM   | `references/{index,patterns,decisions,log}.md` | Operational memory read every session         |

Territory rules:

- The LLM NEVER edits or renames files in `raw/`. Raw is human territory.
- The LLM owns wiki files fully under the `<context>/<category>/<concept>`
  taxonomy with frontmatter validation.
- The LLM owns meta files. `_index.md` is fully auto-regenerated by
  `scripts/lint.sh` between sentinel markers - never hand-edited.
  `patterns.md` / `decisions.md` / `log.md` are append-first operational
  memory that compounds over time.

---

## The Read-Before / Write-After Protocol

### Read BEFORE any task

Every session, in this order:

1. `references/_index.md` - what this skill already knows (the map)
2. `references/patterns.md` - what has worked, with numbers
3. `references/decisions.md` - past reasoning, so you don't repeat mistakes
4. Last 5 entries of `references/log.md` - recent session context
5. Relevant `references/wiki/<context>/<category>/` concepts for the task

No exceptions. The Brain Protocol block in `SKILL.md` is non-negotiable.

### Write AFTER any task

Always:

1. Append a session entry to `log.md`.

When the user provides new material:

2. The raw file lands in `references/raw/` with its original name. The LLM
   does not rename it.
3. Distill into one or more `references/wiki/<context>/<category>/<concept>.md`
   files. Each concept MUST cite the raw file in its `sources:` array.

When the work produces quantified results:

4. Append an entry to `patterns.md` (see §Patterns Entry Shape below).

When the work involves a non-obvious choice:

5. Append an entry to `decisions.md` (Context / Options / Chose / Why / Result).

After any wiki change: run `scripts/lint.sh` to regenerate `_index.md` and
catch path / frontmatter mismatches.

### Patterns Entry Shape

`patterns.md` is where quantified outcomes accumulate. After 20 entries,
the agent stops guessing and starts predicting what works.

**Incorrect (vague, unmeasured, unactionable):**

```markdown
### 2026-04-12 | Content Post
Wrote a LinkedIn post. It did pretty well.
What worked: good hook.
```

No numbers, no hook described, no replicable rule. Future agents can't
learn from it.

**Correct (quantified, specific, action-generating):**

```markdown
### 2026-04-12 | LinkedIn Post: Security Checks Lead Magnet
- Context: test disaster-story hook on a high-stakes security topic
- Approach: disaster story + $87,500 fraud hook + lead magnet CTA (SHIP keyword)
- Result:
  - Impressions: 75,613
  - Comments: 117 (100+ from decision-makers)
  - Engagement rate: 0.31%
- Worked:
  - $87,500 fraud hook stopped the scroll (highest first-3-lines retention in 30 days)
  - SHIP keyword CTA converted 100+ comments with minimal friction
- Didn't:
  - 0.31% engagement is below average; massive reach, most lurked
  - No follow-up DM automation; most commenters weren't contacted for 48h
- Lesson: DISASTER STORY + DOLLAR AMOUNT + LEAD MAGNET CTA = viral formula.
  Replicate hook structure. Add DM automation to capture lurkers.
```

Required fields per entry:

| Field      | Required | Notes                                               |
|------------|----------|-----------------------------------------------------|
| Title line | Yes      | `### <YYYY-MM-DD> \| <short description>`           |
| Context    | Yes      | What was attempted, in one sentence                 |
| Approach   | Yes      | The method / structure / specific choices           |
| Result     | Yes      | Numeric outcomes; include units and baselines       |
| Worked     | Yes      | What specifically helped (evidence-backed)          |
| Didn't     | Yes      | What hurt or under-performed                        |
| Lesson     | Yes      | The rule to apply next time, stated as an imperative|

**When to skip:** if the session produced no measurable outcome, don't
invent one. Write the session to `log.md` and skip `patterns.md`.
Patterns are for actual evidence.

**When to revise:** never edit old entries. If a later result contradicts
an earlier one, add a new entry and reference the older one:

```markdown
### 2026-05-20 | Disaster-hook formula: diminishing returns
- Context: third replication of the 2026-04-12 formula
- Result: impressions down 62% vs first run
- Lesson: formula is fatiguing to the audience. Rotate hooks after 2 uses.
  See 2026-04-12 entry for original result.
```

**Reading `patterns.md`:** every session starts with a full read. If the
file grows past ~100 entries, consider summarising older entries into a
wiki concept under `brain/patterns/` and keeping only the last 30 entries
in the live file. Lint can flag this threshold.

## Three Operations

### Ingest (detailed)

The user drops a file into `references/raw/`. The agent distills it into
one or more wiki concepts and updates the brain. No script required -
this is a pure agent operation.

**Incorrect (shallow ingest - one file in, one file out, no brain update):**

```
User: <drops article.md into references/raw/>
Agent: writes references/wiki/content/article.md with a summary
Agent: done
```

No `sources:` cited, no `log.md` entry, no category/concept split, no
`last_ingested` date, no lint run. After 20 ingests the brain has no
idea what it knows.

**Correct (full ingest):**

```
User: <drops references/raw/karpathy-brain-idea.md>

Agent:
1. Reads references/_index.md, patterns.md, decisions.md, last 5 of log.md
2. Reads references/raw/karpathy-brain-idea.md
3. Identifies 3 distinct concepts spanning 2 contexts:
   - brain/protocol/three-layer-architecture
   - brain/patterns/how-to-log-results
   - smart/structure/directory-layout (contributes to existing concept)
4. For each new concept, writes references/wiki/<ctx>/<cat>/<concept>.md
   with frontmatter:
     context: <ctx>
     category: <cat>
     concept: <concept>          # matches filename stem
     sources:
       - "references/raw/karpathy-brain-idea.md"
     last_ingested: 2026-04-16
5. Adds karpathy-brain-idea.md to the sources: of the existing
   smart/structure/directory-layout concept
6. Appends to references/log.md:
     [INGEST 2026-04-16] Processed karpathy-brain-idea.md.
       Produced 2 new concepts, updated 1.
7. If the material includes metrics, appends to patterns.md
8. If splitting concepts required a non-obvious call, appends to decisions.md
9. Runs scripts/lint.sh to regenerate _index.md and verify
```

Step-by-step:

1. **Read the brain** (mandatory - Brain Protocol; skip = regressions).
2. **Read the raw file** in full. If large, read in chunks but process
   the entire file.
3. **Decide scope.** Ask: "How many distinct concepts does this file
   seed?" A single raw file often spans multiple contexts/categories.
   Do NOT cram multiple concepts into one wiki file - split.
4. **For each concept:**
   - Exists already? Update the concept file in place. Add the raw path
     to its `sources:` if not already listed. Refresh `last_ingested`.
   - New concept? Create `references/wiki/<context>/<category>/<concept>.md`
     with full frontmatter + body per `references/_template.md`.
5. **Write log.md entry** (always). At minimum:
   ```
   [INGEST <YYYY-MM-DD>] Processed <raw-file>.
     Produced <N> new concepts, updated <M>.
   ```
6. **Conditional writes:** metrics -> `patterns.md`; non-obvious
   splits/merges -> `decisions.md`.
7. **Lint:** `bash scripts/lint.sh`. Regenerates `_index.md`, validates
   path/frontmatter triples, flags orphans and broken sources. Fix
   findings before finishing.

### Query

Brain Protocol IS the operation. Read brain first, synthesize from wiki,
optionally write novel concepts back.

### Lint Checks

`scripts/lint.sh` is the brain's health check. Run every 2 weeks, before
releases, or after any wiki change. Exit code is 0 only when all hard
checks pass.

The taxonomy (valid `<context>`/`<category>` pairs) is **derived from the
filesystem** - there is no registry file. Any `<context>/<category>` path
that exists under `references/wiki/` is valid by definition.

Checks performed:

1. **Path / frontmatter triple match.** Every wiki file at
   `references/wiki/<a>/<b>/<c>.md` MUST have frontmatter `context: <a>`,
   `category: <b>`, `concept: <c>`.
   ```
   MISMATCH: references/wiki/brain/lint/checks.md
     path:   context=brain, category=lint, concept=checks
     front:  context=brain, category=lint, concept=lint-checks
   ```
   Fix: rename the file OR change `concept:` so both agree.

2. **Required frontmatter.** Every wiki file MUST have: `title`,
   `context`, `category`, `concept`, `description`, `tags`,
   `sources`, `last_ingested`.
   ```
   MISSING FIELDS: references/wiki/brain/ingest/workflow.md
     missing: last_ingested
   ```
   Fix: add the missing field.

3. **Orphan raw files.** Raw files not cited by any wiki concept's
   `sources:` array.
   ```
   ORPHAN RAW: references/raw/old-dump.md
     not cited by any wiki concept
   ```
   Fix: (a) ingest into a wiki concept that cites it in `sources:`, or
   (b) delete the raw file if it's no longer needed.

4. **Broken sources.** `sources:` paths in wiki frontmatter that don't
   resolve to a real file under `references/raw/`.
   ```
   BROKEN SOURCE: references/wiki/content/hooks/disaster.md
     sources[0]: references/raw/does-not-exist.md
   ```
   Fix: correct the path OR remove the entry from `sources:`.

5. **Orphan wiki concepts.** Wiki concepts with empty `sources:` array.
   Warning, not error - sometimes a concept is pure synthesis with no
   raw origin. Flagged so you can audit.

6. **Stale entries.** Wiki files whose `last_ingested` is older than the
   configured threshold (default: 180 days, override via `STALE_DAYS`
   env var). Surfaces candidates for review; doesn't fail.

7. **Contradictions (heuristic).** Two wiki files sharing the same
   `concept:` value (even under different paths). Often a legitimate
   naming reuse (e.g. `workflow` across multiple categories); flagged
   as WARN for human audit.

**Side effect - `_index.md` regeneration.** `lint.sh` rewrites everything
between `<!-- GENERATED:START -->` and `<!-- GENERATED:END -->` markers
in `references/_index.md`. Emits, in order:

1. `## Summary` - context -> categories TOC (no concepts)
2. `## Wiki` - full `### context` / `#### category` / concept-bullet tree
3. `## Raw Sources` - raw files + their citers
4. `## Scripts` - script enumeration

Never hand-edit the generated region - edit the wiki frontmatter and
re-run lint.

**Exit codes:**

| Code | Meaning                                              |
|------|------------------------------------------------------|
| 0    | All hard checks passed                               |
| 1    | One or more hard findings (mismatch, missing, broken)|
| 2    | Invocation error (wrong CWD, missing files)          |

## Naming Conventions

### Contexts and categories (the taxonomy)

- `context` - first path segment under `wiki/`; 3-10  chars, lowercase, no
  hyphens (e.g. `smart`, `brain`, `query`, `schema`).
- `category` - second path segment; kebab-case, 1-3 words (e.g. `create`,
  `linkedin-hooks`, `directory-layout`).
- The taxonomy is **derived from the filesystem**. `scripts/lint.sh` walks
  `references/wiki/` to build the context -> categories map. Move a file,
  the taxonomy updates. There is no separate registry file to drift.

### Concepts (the leaves)

- `concept` - filename stem of a wiki file AND a required frontmatter
  field. Kebab-case, descriptive, 2-5 words.
- One concept per file. If a single dump seeds three distinct concepts,
  that's three wiki files.
- The `context` / `category` / `concept` triple in frontmatter MUST match
  the filesystem path `references/wiki/<ctx>/<cat>/<concept>.md`.
  `lint.sh` enforces this.

```
# Good
references/wiki/smart/create/workflow.md
references/wiki/brain/ingest/workflow.md
references/wiki/content/linkedin-hooks/disaster-hook-formula.md

# Bad
references/wiki/s/create/workflow.md              # context too short
references/wiki/smart/Create/workflow.md          # category not kebab-case
references/wiki/smart/create/the_workflow.md      # underscore in concept
references/wiki/smart/create/stuff.md             # vague concept
```

### Raw files

Human territory. No rules. Keep whatever filename the human provided -
spaces, caps, any extension. The LLM reads but never renames or edits.

## Wiki `sources:` Contract

Every wiki concept carries a `sources:` array in its frontmatter. This is
the ONLY authoritative link from distilled knowledge back to raw material.

```yaml
sources:
  - "references/raw/2026-04-12 linkedin analytics.csv"
  - "references/raw/notes from call.md"
```

Rules:

1. Paths are relative to the skill root. Always start with `references/raw/`.
2. Always quoted. Raw filenames may contain spaces or special characters.
3. Must resolve. `lint.sh` flags broken paths.
4. Many-to-many: one raw file may appear in many concepts' `sources:`;
   one concept may cite many raw files.
5. Empty list is legal but flagged as orphan (pure synthesis concepts).
6. Never edit raw to satisfy a wiki claim. If a concept over-claims, fix
   the concept.

The reverse map (raw -> citing concepts) is regenerated into `_index.md`
by `lint.sh` - not hand-maintained.

## Writing Principles

- **Show concrete transformations** - "Change X to Y", not abstract advice
- **Error-first structure** - show the problem before the solution
- **Quantify impact** - include specific metrics ("50% fewer tokens",
  "3x faster") where possible
- **Self-contained examples** - complete and copy-pasteable; no `...`
  elisions in critical paths
- **Semantic naming** - use meaningful names in examples, not `foo`/`bar`
- **One concept per file** - if covering two concerns, split into two files
- **Cite raw sources** - populate `sources:` honestly, not decoratively
- **Challenge verbosity** - every paragraph must justify its token cost

---

## Command Routing

Wiki concepts may declare a `command:` in frontmatter pointing at
`scripts/<name>.sh|.py`. Scripts must be deterministic, non-interactive,
idempotent, and self-contained. Multiple concepts may share the same script.

---

## Why This Works

- Raw stays immutable so the user always has ground truth.
- Wiki stays distilled so context loading is cheap (on-demand per concept).
- Meta stays operational so the agent walks in knowing what happened last time.
- `sources:` provides the causal chain from every claim back to its origin.
- Lint keeps the system self-consistent as it grows, and the taxonomy
  is derived from the filesystem so there's no registry to drift.

The compounding mechanism: every session writes at least one line to
`log.md`. Every outcome with numbers writes a line to `patterns.md`.
Every choice worth remembering writes a line to `decisions.md`. After 30
sessions, the agent starts every task with 30 sessions of context.

No data, no improvement.
